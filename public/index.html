basic structure of MCP

An MCP (Model Context Protocol) setup typically consists of a client and one or more servers. The client initializes a session, negotiates capabilities, and invokes tools/resources exposed by servers. Servers declare metadata, available tools, and resources; they receive JSON-RPC style requests, execute the work, and stream results, logs, and errors back. Common transports include stdio, HTTP, or WebSocket, with a simple handshake, request/response IDs, and structured error reports.

example MCP Servers

Examples include: Neon (Postgres DB), Netlify (hosting/deploy), Zapier (automation), Figma (design to code via plugin), Builder CMS (content), Linear (issues), Notion (docs), Supabase (auth/DB), Sentry (errors), Context7 (docs), Semgrep (security), and Prisma Postgres (ORM). These servers expose actions and resources the client can call to orchestrate workflows end‑to‑end.

Pitfalls developing MCP

Common pitfalls: weak schema/versioning between client and server; blocking or long‑running tools with no timeouts; missing retries/backoff; leaking secrets in logs; poor auth refresh and token storage; rate‑limit handling and idempotency gaps; inadequate streaming/backpressure control; unclear error codes; and lack of local tests/mocks. Prefer explicit schemas, timeouts, retries, structured errors, and secure secret management.
