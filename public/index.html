basic structure of MCP

An MCP (Model Context Protocol) setup typically consists of a client and one or more servers. The client initializes a session, negotiates capabilities, and invokes tools/resources exposed by servers. Servers declare metadata, available tools, and resources; they receive JSON-RPC style requests, execute the work, and stream results, logs, and errors back. Common transports include stdio, HTTP, or WebSocket, with a simple handshake, request/response IDs, and structured error reports.

Pitfalls developing MCP

Schema/version mismatch

Divergent versions between client and server cause malformed requests or silently ignored fields. Example: client sends tool parameters added in v2 while server still validates against v1; add explicit version fields and schema negotiation. Related CVEs: none specific; impact varies by implementation.

No timeouts for long‑running tools

Unbounded executions tie up workers and block queues. Example: external API hang without deadline; add per‑tool timeouts, cancellations, and circuit breakers. Related CVEs: none specific; typically availability issues.

Missing retries/backoff

Transient failures escalate into outages if calls aren’t retried conservatively. Example: apply exponential backoff with jitter for 5xx and network errors, and cap attempts. Related CVEs: none; resilience practice.

Secrets in logs

Tokens, API keys, or PII in logs risk exfiltration. Example: redact Authorization headers and env vars; use allow‑lists for log fields. Related CVEs: many incidents exist but are product‑specific; avoid logging secrets entirely.

Auth refresh/storage issues

Poor token rotation or plaintext storage leads to session hijack. Example: store tokens in KMS/secret manager, rotate short‑lived creds, and pin scopes. Related CVEs: product‑specific across OAuth providers; follow least‑privilege and rotation.

Rate limits and idempotency

Ignoring 429s or using non‑idempotent operations on retry creates duplicate side effects. Example: use idempotency keys and respect Retry‑After headers. Related CVEs: not applicable; correctness/abuse control.

Streaming/backpressure

Unbounded streams exhaust memory or sockets. Example: apply flow control, chunked processing, and consumer‑driven pull. Related CVEs: none specific; stability concern.

Unclear error codes

Ambiguous errors hinder recovery logic. Example: structured errors with codes like invalid_argument, unauthenticated, deadline_exceeded. Related CVEs: not applicable.

Lack of local tests/mocks

Without mocks, regressions slip into production. Example: simulate server tools/resources with fixtures and golden responses in CI. Related CVEs: not applicable; quality practice.
